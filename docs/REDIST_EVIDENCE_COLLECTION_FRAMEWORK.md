# ReDist Evidence Collection Framework

Date: 2026-06-22  
Status: Evidence collection framework for real pilot transfers  
Purpose: Capture, validate, and organize proof for future case studies, impact stories, and sales assets.

## Evidence Principle

ReDist should publish only evidence that is:

- Connected to a completed transfer.
- Reviewed for accuracy.
- Approved by relevant organizations.
- Redacted for privacy and confidentiality.
- Clearly labeled if estimated or unaudited.

## Evidence Stack

| Evidence Layer | Purpose | Required For Case Study |
| --- | --- | --- |
| Listing screenshot | Shows what resource was offered | Yes |
| Request screenshot | Shows recipient request | Yes |
| Transfer reference | Shows workflow progression | Yes |
| Approval record | Shows supplier decision | Yes |
| Verification record | Shows completion signal | Yes |
| Certificate | Shows operational evidence | Yes |
| Feedback form | Shows user value and issues | Yes |
| Founder observations | Shows process learning | Yes |
| Photos | Shows real-world context | Optional with consent |
| Impact calculation notes | Explains value/waste estimates | Required if metrics are published |

## Screenshots

Capture screenshots for:

- Listing detail.
- Request detail.
- Transfer status.
- Certificate view.
- Impact summary.
- Dashboard or workflow queue if relevant.

Screenshot rules:

- Redact names, emails, phone numbers, private notes, internal IDs, and sensitive organization details unless approved.
- Store original internal screenshots separately from public-safe versions.
- Do not publish screenshots from protected staging without approval.

## Certificates

Collect:

- Certificate reference.
- Certificate PDF or image.
- QR/public verification result if applicable.
- Certificate date.
- Supplier and recipient names as approved.
- Resource and quantity.

Certificate rules:

- Confirm certificate matches completed transfer.
- Confirm public-safe data.
- Explain certificate limitations before using externally.
- Do not present certificate as legal, tax, safety, government, or audited ESG proof.

## Transfer References

Collect:

- Listing reference.
- Request reference.
- Transfer reference.
- Certificate reference.
- Handover date.
- Completion date.
- Supplier decision status.
- Recipient verification status.

Reference rules:

- Use internal references for auditability.
- Use public-friendly references only after review.
- Do not expose sensitive internal IDs if not intended for public use.

## Verification Records

Collect:

- Organization verification status.
- Approved category.
- Handover verification notes.
- Completion confirmation.
- Any dispute or exception record.

Verification rules:

- Verification records support trust context.
- They do not imply ReDist inspected item quality or guaranteed compliance.
- Restricted category decisions require founder/legal review.

## Feedback Forms

Collect:

- Supplier feedback.
- Recipient feedback.
- Ease of use rating.
- Trust rating.
- Certificate usefulness rating.
- Impact usefulness rating.
- Problems encountered.
- Permission to quote.
- Permission to name organization.

Use:

- [UAE Pilot Feedback Form](UAE_PILOT_FEEDBACK_FORM.md)

Feedback rules:

- Do not quote participants without approval.
- Do not publish private criticism without redaction.
- Capture negative feedback as product learning, not marketing copy.

## Founder Observations

Collect:

- What required founder help?
- What caused confusion?
- What risk appeared?
- What slowed handover?
- What the participant valued most?
- What should change before the next pilot?

Founder observation rules:

- Separate observation from participant quote.
- Label interpretation clearly.
- Include observations in post-mortem and case study draft.
